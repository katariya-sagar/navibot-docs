API direction

Early interface shape

These endpoints describe the first useful boundary between a client interface and route intelligence service.

Candidate endpoints

  • POST /trips/plan: parse a request and return route recommendations.
  • POST /trips/compare: compare route candidates against user constraints.
  • POST /feedback: record whether a recommendation helped.
  • GET /health: expose service health for deployment checks.

Example request

{
  "origin": "Airport Terminal 2",
  "destination": "City Center",
  "departureTime": "2026-06-26T09:00:00+05:30",
  "preferences": ["fewest_transfers", "reliable"]
}

Example response

{
  "recommendation": {
    "summary": "Take the express rail route",
    "estimatedMinutes": 38,
    "reasons": ["fastest", "low transfer risk", "short walk"]
  },
  "alternatives": [
    { "summary": "Taxi", "estimatedMinutes": 46 },
    { "summary": "Bus plus metro", "estimatedMinutes": 55 }
  ]
}

Design notes

Keep provider-specific fields behind adapters. The API should return stable NaviBot objects even when the map or routing provider changes.
